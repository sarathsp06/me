<SectionHeading icon="fa-lightbulb" color="amber">The Problem</SectionHeading>

<div class="prose-content">

If you have ever integrated a third-party service that sends webhooks -- payment providers, CI/CD pipelines, messaging platforms -- you know the pain. You need a publicly accessible URL to receive the callback, a way to inspect the payload, and some mechanism to forward it to your local development server.

The typical options are:

- **webhook.site** -- great, but it is a SaaS. Your payloads go through someone else's infrastructure. For many teams, that is a non-starter.
- **ngrok** -- exposes your local port to the internet, but requires a CLI daemon, often has rate limits, and still does not give you a persistent capture UI.
- **RequestBin** -- the original was shut down. Alternatives exist but are often unmaintained or over-engineered.

I wanted something simpler: a tool I could self-host, run on my own infrastructure, and trust with sensitive webhook payloads. That is why I built **Testhooks**.

</div>

<SectionHeading icon="fa-rocket" color="blue">What Testhooks Does</SectionHeading>

<div class="prose-content">

Testhooks gives you unique webhook URLs. Any HTTP request sent to those URLs is captured and streamed to your browser in real time via WebSocket. From there, you can inspect headers, body, query parameters -- everything.

</div>

```
webhook sender --> https://hooks.example.com/h/a1b2c3d4 --> your browser (live)
```

<SectionHeading icon="fa-toggle-on" color="emerald">Two Modes: Server vs. Browser</SectionHeading>

<div class="prose-content">

Testhooks has two fundamentally different endpoint modes, and choosing the right one depends on your use case.

</div>

<div class="grid grid-cols-1 md:grid-cols-2 gap-4 my-6">
  <div style="background: linear-gradient(135deg, #f0fdf4, #f7fef9); border: 1px solid #bbf7d0; border-radius: 10px; padding: 1.5rem;">
    <div class="flex items-center gap-3 mb-3">
      <span class="w-7 h-7 rounded-full bg-emerald-600 text-white flex items-center justify-center text-xs font-bold"><i class="fas fa-server"></i></span>
      <h3 class="text-lg font-semibold text-gray-900">Server Mode</h3>
    </div>
    <div class="prose-content">
      <p class="mb-0">Requests are stored in PostgreSQL. Transforms and forwards run headlessly on the Go server -- even when no browser is open. Ideal for <strong>always-on integrations</strong> and CI pipelines.</p>
    </div>
  </div>

  <div style="background: linear-gradient(135deg, #eff6ff, #f8faff); border: 1px solid #bfdbfe; border-radius: 10px; padding: 1.5rem;">
    <div class="flex items-center gap-3 mb-3">
      <span class="w-7 h-7 rounded-full bg-blue-600 text-white flex items-center justify-center text-xs font-bold"><i class="fas fa-globe"></i></span>
      <h3 class="text-lg font-semibold text-gray-900">Browser Mode</h3>
    </div>
    <div class="prose-content">
      <p class="mb-0">Zero storage. Payloads never touch the server's disk -- all processing happens client-side in the browser. Perfect for <strong>sensitive payloads</strong> and quick debugging sessions.</p>
    </div>
  </div>
</div>

<Callout type="insight">
  <p class="font-semibold text-gray-800 text-sm mb-1">Why Browser Mode Matters</p>
  <p class="text-sm text-gray-600">Browser mode is what sets Testhooks apart from most alternatives. When you are debugging a payment webhook that contains PII or financial data, you do not want that payload stored on a server -- not yours, and definitely not a third party's. Browser mode streams the raw request over WebSocket and processes everything in the browser. When you close the tab, the data is gone.</p>
</Callout>

<SectionHeading icon="fa-play-circle" color="indigo">Getting Started</SectionHeading>

<div class="prose-content mb-6">

The fastest way to get Testhooks running is Docker Compose. It starts the application and a PostgreSQL instance together.

</div>

<StepList>
  <StepItem number={1} title="Clone and start">
    <div class="prose-content">
      <p class="text-sm text-gray-600 leading-relaxed">Clone the repository and start everything with a single command:</p>
    </div>
    ```bash
    git clone https://github.com/sarathsp06/testhooks.git
    cd testhooks
    docker compose up --build
    ```
  </StepItem>

  <StepItem number={2} title="Create an endpoint">
    <div class="prose-content">
      <p class="text-sm text-gray-600 leading-relaxed">Open <code>http://localhost:8080</code> in your browser. Create a new endpoint -- you will get a unique URL like <code>/h/a1b2c3d4</code>. Choose Server mode or Browser mode depending on your needs.</p>
    </div>
  </StepItem>

  <StepItem number={3} title="Send a webhook">
    <div class="prose-content">
      <p class="text-sm text-gray-600 leading-relaxed">Point your webhook sender at the generated URL. The request will appear in your browser instantly.</p>
    </div>
    ```bash
    curl -X POST http://localhost:8080/h/a1b2c3d4 \
      -H "Content-Type: application/json" \
      -d '{"event": "payment.completed", "amount": 42.00}'
    ```
  </StepItem>

  <StepItem number={4} title="Inspect, transform, forward">
    <div class="prose-content">
      <p class="text-sm text-gray-600 leading-relaxed">View full request details (headers, body, query params). Optionally write a transform script to reshape the payload, or set up forwarding to relay it to your local dev server.</p>
    </div>
  </StepItem>
</StepList>

<div class="prose-content">

The single-binary design is intentional. There are no static files to serve separately, no separate frontend deployment, no CDN configuration. Build the binary, point it at PostgreSQL, and it works. This makes it trivial to deploy on any PaaS (Heroku, Railway, Render) or just run it on a VPS behind a reverse proxy.

</div>

<SectionHeading icon="fa-shield-alt" color="red">Replacing ngrok for Webhook Development</SectionHeading>

<div class="prose-content">

One of the most common use cases for ngrok is webhook development: you need a public URL that tunnels to your local machine so third-party services can reach your development server. Testhooks replaces this workflow without requiring a CLI or daemon.

Here is how it works:

1. Deploy Testhooks to a public server (or use the hosted instance at [testhooks.sarathsadasivan.com](https://testhooks.sarathsadasivan.com/)).
2. Create an endpoint and point your webhook sender at it.
3. In the Testhooks UI, set up **browser-side forwarding** to `http://localhost:3000` (or wherever your dev server runs).
4. When a webhook arrives, your browser receives it via WebSocket and immediately `fetch()`es it to your local server.

No tunnel, no port exposure, no CLI running in the background. The browser acts as the bridge between the public internet and your local machine.

</div>

<Callout type="tip" title="When to Use Server-Side Forwarding" icon="fa-server">
  <p class="text-sm text-gray-600">Browser-side forwarding only works when your browser tab is open. For production-like scenarios or CI pipelines, use server-side forwarding instead -- it runs headlessly on the Go server and reliably forwards to any public URL, even when no browser is connected.</p>
</Callout>
